# 一个计算机能维护的最大连接数由什么决定？

## 从 accept 队列说起

`accept()` 从全连接队列取走一个已完成三次握手的 socket，把它交给用户程序使用（分配一个 fd）。但这之后的连接，瓶颈不在 accept 队列，而在操作系统资源。

## 核心限制因素

### 1. 文件描述符上限（最直接的）

```bash
ulimit -n  # 每个进程最多能打开多少个 fd
# 默认 1024，可以改到 65535 甚至百万级
```

每条 TCP 连接占用一个 socket fd。`ulimit -n = 1024` 意味着包括标准输入输出、文件、监听 socket 在内，最多 1024 个 fd——留给连接的只有 1020 个左右。

### 2. 内存（真正的硬限制）

每个 socket 内核对象占用约 **1-3 KB** 内核内存（不包括应用层缓冲区）：

```
100 万连接：
  1MB 每个 → 1 TB   ❌
  3KB 每个 → 3 GB   ✓（现代服务器可以）
```

加上读写缓冲区：

```
读缓冲区默认 87 KB
写缓冲区默认 16 KB
每个连接: ~100 KB

100 万连接 × 100 KB = 100 GB ❌
```

所以优化高并发服务器必须调小缓冲区：

```rust
let socket = TcpStream::new()?;
socket.set_recv_buffer_size(4096)?;  // 调小
socket.set_send_buffer_size(4096)?;  // 调小
```

### 3. 内核跟踪表

内核维护多个全局表，每个连接占一项：

| 表 | 条目数上限 | 说明 |
|---|-----------|------|
| `tcp_mem` | 按物理内存计算 | 总 TCP 内存上限 |
| `tcp_max_tw_buckets` | 默认 18 万 | TIME_WAIT 状态上限 |
| conntrack | 默认 65536 | 防火墙连接跟踪 |

### 4. epoll 性能

```bash
cat /proc/sys/fs/epoll/max_user_watches
# epoll 能监控的最大 fd 数，每个 fd 约 90 字节
```

### 5. 大致数量级

```
                      个位数 GB 内存                    几十 GB 内存
                     ─────────────                    ─────────────
普通服务器:            ~5 万连接                        ~50 万连接
高性能调优（C/Rust）:   ~30 万连接                       ~200 万-1000 万连接
```

## 实际案例：C10K → C10M

| 年代 | 目标 | 瓶颈 |
|------|------|------|
| 1999 | C10K（1 万） | 线程/进程模型，一个连接一个线程 |
| 2010 | C100K（10 万） | epoll/kqueue 解决了 |
| 2013 | C1M（100 万） | 内存 + 内核参数调优 |
| 2015 | C10M（1000 万） | 内存成为主要瓶颈 |

1000 万连接需要：
- 每连接 3KB 内核内存 → 30 GB
- 每连接 4KB 应用缓冲区 → 40 GB
- 总共 **70+ GB 内存**

## accept 队列满 vs fd 耗尽

这是两个独立的瓶颈：

| 瓶颈 | 标志 | 解决 |
|------|------|------|
| **accept 队列满** | `accept()` 返回错误 | 增大 backlog + 加快处理速度 |
| **fd 耗尽** | `accept()` 返回 `EMFILE` | 增大 ulimit + 减少空闲连接 |

即使 `ulimit -n` 大到百万，accept 队列还是可能满——backlog 限制死了排队上限，处理速度跟不上到达速度时队列就满了。

## 一句话总结

**一个计算机能维护的最大连接数 ≈ 内存大小 ÷ 每个连接的开销。** 在有足够内存的前提下，`ulimit -n` 可以调大，`tcp_mem` 可以调大，真正的物理限制是内存装不下更多连接的内核对象和缓冲区了。
